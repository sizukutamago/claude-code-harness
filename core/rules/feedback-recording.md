# フィードバック記録ルール

## 必須

ユーザに指摘・修正・方向転換を受けたら、即座に `.harness/session-feedback.jsonl` に記録しろ。

「後でまとめて記録する」な。指摘された瞬間に書け。

## 記録すべき場面

- ユーザが「それは違う」「そうじゃない」と修正した
- ユーザが方針・設計・スコープを変更した
- ユーザが「なんで？」と前提を問い直した
- ユーザが代替案を指示した
- ユーザがツール実行を拒否した

## 記録しないもの

- ユーザの通常の指示（「次はXをやって」等）
- ユーザの承認（「OK」「いいよ」等）
- ユーザの質問への回答

## 記録フォーマット

`.harness/session-feedback.jsonl` に1行追記する:

```json
{"id":"fb-XXX","timestamp":"YYYY-MM-DDTHH:MM:SSZ","status":"open","type":"correction","category":"scope|spec|assumption|design|naming","summary":"何を指摘されたかの要約","user_said":"ユーザの発言（原文をできるだけ忠実に）","affected":"反映先のファイルパス","session_id":"セッションID"}
```

- `id`: `fb-` + 連番（既存エントリの最大値 + 1）
- `status`: 常に `open`
- `type`: `correction`（ユーザによる修正指摘）
- `category`: 以下のいずれか
  - `scope` — 責務の越境（「それはXの仕事」）
  - `spec` — 仕様・入力の漏れ（「Yが足りない」）
  - `assumption` — 前提のミス（「なんでZを使うの？」）
  - `design` — 過剰設計・判断の誤り（「それは重すぎる」）
  - `naming` — 命名・用語の不適切（「Aじゃなくて B にして」）

## 禁止

- フィードバックの記録を省略するな
- `status` を `open` 以外にするな（遷移は retrospective の責務）
- 記録の際にユーザに確認を求めるな（記録するかどうかは自分で判断しろ）
- 記録したら「📝 フィードバック記録: [要約]」と1行だけ表示しろ（動作確認のため）
