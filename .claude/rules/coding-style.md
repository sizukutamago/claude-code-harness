# Coding Style Rules

## 原則

**読みやすさは書きやすさに勝る。**

プロジェクトに lint/formatter の設定（ESLint, Biome, Prettier 等）がある場合はそれに従うこと。このルールよりプロジェクトの lint 設定が優先される。

## 必須ルール

1. **意図を表す命名**: 変数名・関数名は省略せず意図を表現する（`usr` ではなく `user`）
2. **イミュータビリティ優先**: 変更可能な状態を最小限にする。再代入より不変を選ぶ
3. **早期リターン**: ネストを減らすために guard clause を使う
4. **DRY**: 同じロジックが3箇所以上に現れたら共通化する。2箇所の時点では早すぎる抽象化を避ける

## 禁止事項

- 無関係な機能を1ファイルにまとめる（ユーザー認証とメール送信を同じファイルに書く等）
