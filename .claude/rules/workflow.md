# Workflow Rules

## ワークフロー開始

`/start-workflow` でワークフローを開始する。ユースケース（機能開発・バグ修正・ドキュメント更新・テスト修正・リファクタリング・設定変更）を選択すると、適切なステップが自動で進行する。

## ワークフロー（12ステップ）

```
[1] 要件理解 → [2] 設計 → [3] 計画
→ [4] 実装 → [5] テスト → [6] リファクタ → [7] 品質テスト → [8] レビュー
→ [9] 完了検証 → [10] 整理 → [11] コミット → [12] 振り返り
```

### タスク規模別の適用ルール

全12ステップを毎回適用する必要はない。タスクの規模に応じてデフォルトパスが異なる。

| 規模 | 判定基準 | 適用ステップ |
|------|---------|------------|
| **Tiny** | typo修正、設定値変更、1行の修正 | [4] 実装 → [9] 検証 → [11] コミット |
| **Small** | **単一の関心事（single concern）** を持つ変更。ファイル数ではなく「1つの目的で束ねられるか」で判定する | [4][5] 実装・テスト → [8] レビュー → [9] 検証 → [11] コミット（[4] と [5] は tdd スキルが一体で実行する） |
| **Normal** | 複数ファイル・新機能・設計判断を伴う変更 | 全12ステップ |
| **Large** | 複数機能・フェーズ分割が必要・2スプリント以上 | roadmap → (planning → tdd → ... → commit) × N → [12] 振り返り |

**運用ルール:**
- 規模の判定はセッション開始時に人間パートナーに確認する
- 迷ったら Normal（全ステップ）を適用する
- Tiny/Small でも Invariants（不変制約）は常に適用される
- Large の場合、roadmap スキルでフェーズ分割してから各フェーズを Normal ワークフローで回す
- Large の [12] 振り返りは全フェーズ完了後に1回だけ実行する（各フェーズ後には実行しない）

### 複数の独立タスクを1セッションで扱う場合

独立した改善提案・バグ修正を連続で処理する場合は、以下のルールに従う:

- セッション冒頭で全タスクを列挙し、**各タスクごと** に Tiny/Small/Normal の規模を判定する
- セッション全体を1つの規模に強制してはいけない。各タスクは独立したワークフロー実行として扱う
- セッション末尾の [12] 振り返りは、個別タスクごとではなく **セッション末で1回だけ** 実行する（Large と同様の運用）
- 各タスクの [9] 完了検証は `last-verification.json` を都度更新する（verification-gate フックが古い検証証拠をブロックする）

## Invariants（不変制約）

以下はプロジェクト設定で変更できない。常に適用される。

- 検証証拠なしに完了を宣言しない
- 要件を推測・捏造しない（必ず人間に確認）
- 破壊的・不可逆操作は人間の承認が必要
- 振る舞いの変更には実行可能な検証が必要
- 本番環境への直接操作禁止
- シークレットのハードコード禁止
- メインセッションはコードを書かない
- レビュー指摘への対応は人間パートナーの承認後に実行する

## Policies（現在は固定値、将来的に調整可能にする予定）

以下は現在すべて固定値で動作する。設定ファイルによるカスタマイズは未実装。

- 要件定義の深さ: **常時必須**（requirements.md + context.md）
- テストケース設計の分離: **AC を起点に implementer が追加テスト可**
- レビューのトポロジー: **3観点並列**
- simplify の実行方式: **専用エージェントによる別ステップ**
- 人間承認のタイミング: **要件確定時 + 設計承認時 + 計画承認時 + レビュー指摘対応時 + コミット前**

## State Machine

12ステップワークフローのフォーマルな状態マシン定義。

### ステップの状態

| 状態 | 意味 |
|------|------|
| `pending` | 未着手 |
| `in_progress` | 実行中 |
| `completed` | 正常完了 |
| `failed` | 失敗（リトライまたはエスカレーション） |
| `skipped` | タスク規模によりスキップ（Tiny/Small パス） |
| `suspended` | 人間承認待ち |

### 有効な状態遷移

| 遷移元 | 遷移先 | トリガー |
|--------|--------|---------|
| `pending` | `in_progress` | ステップ開始 |
| `pending` | `skipped` | Tiny/Small パスで該当ステップを飛ばす |
| `in_progress` | `completed` | 正常完了 |
| `in_progress` | `failed` | エージェント失敗 |
| `in_progress` | `suspended` | 人間承認ゲート到達 |
| `suspended` | `in_progress` | 人間が承認 |
| `suspended` | `failed` | 人間が却下 |
| `failed` | `in_progress` | リトライ |
| `failed` | `suspended` | リトライ上限到達、人間にエスカレーション |

### サスペンションポイント（人間承認ゲート）

| ステップ | 承認対象 |
|---------|---------|
| [1] 要件確定 | requirements.md の内容・範囲の承認 |
| [2] 設計承認 | design.md のセクション承認 |
| [3] 計画承認 | plan.md のタスク分解の承認 |
| [8] レビュー後 | レビュー指摘（MUST/SHOULD）への対応方針の承認。指摘をそのまま修正に着手しない |
| [11] コミット前 | 変更全体のサマリ + diff の最終承認 |

### 失敗時の遷移先

| 失敗ステップ | 遷移先 |
|------------|--------|
| [4]-[7] 実装・テスト・リファクタ・品質テスト | 同ステップをリトライ |
| [8] レビュー失敗（MUST 指摘あり） | [4] 実装に戻る |
| [9] 検証失敗 | 不足箇所に応じて [4] or [2] に戻る |

## Safety Limits

ワークフロー実行の安全装置定義。

| 制限 | 値 | 超過時のアクション |
|------|---|-------------------|
| 同一ステップのリトライ上限 | 3回 | 人間にエスカレーション（`suspended` 状態に遷移） |
| レビュー→修正ループ上限 | 2周 | 人間にエスカレーション |
| 連続エージェント失敗 | 3回 | ワークフロー一時停止、人間に状況報告 |

**補足:**
- Safety Limits は Invariants と異なり、プロジェクト設定で調整可能にする予定（将来）
- 上限に達した場合、自動的に失敗にせず必ず人間にエスカレーションする
- ログ: エージェント dispatch は workflow-events.jsonl に記録される（観測用。リトライ回数の自動カウントは将来拡張）
