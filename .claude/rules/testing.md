# Testing Rules

## Iron Law

**テストなしにプロダクションコードを書くな。**

## 必須ルール

1. **テストファースト**: RED（失敗するテストを書く）→ GREEN（テストを通す最小コード）→ REFACTOR（テストGREENのまま改善）の順序を守る
2. **変更にはテストが伴う**: 既存コードの変更時、変更対象のテストが存在することを確認してから変更する
3. **全テストGREEN維持**: コミット前に全テストが通ること。失敗テストを放置しない
4. **振る舞いをテストする**: 実装の内部詳細ではなく、外部から観測可能な振る舞いを検証する
5. **テストの独立性**: 各テストは他のテストに依存しない。実行順序に依存しない
6. **バグ修正は再現テストから**: バグを再現するテストを先に書き、失敗を確認してから修正する

## 禁止事項

- `console.log` デバッグだけで問題を確認し、テストを書かずに終わらせる
- テストをスキップ（`skip`, `xit`, `xtest`）したままコミットする
- カバレッジを稼ぐだけで仕様を検証していないテストを書く（`expect(true).toBe(true)` のような常に通るアサーション等）
- モックに依存しすぎて、実際の入出力や副作用を一切検証していないテストを書く
- テスト内で条件分岐（`if`）を使う（テストケースを分割せよ）

## テスト設計

- テスト名は「何が」「どういう条件で」「どうなるか」を記述する
- 正常系・異常系・境界値の3観点でテストケースを設計する
- カバレッジ数値の達成ではなく、振る舞いの網羅を目的にする
