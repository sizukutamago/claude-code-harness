# エラー分類

エージェント実行中にエラーが発生した場合の分類基準と対応アクションを定義する。

## 3分類

### Fatal（致命的 — 人間介入必須）

自動リトライでは解決不可能。人間パートナーへのエスカレーションが必要。

判定基準:
- 要件の理解齟齬（仕様の解釈が人間の意図と異なる）
- スコープ違反（requirements.md の「やらないこと」に該当する実装）
- セキュリティ問題（secret-scanner がブロックした内容）
- 設計の根本的な問題（アーキテクチャ変更が必要）

対応アクション:
- 即座に作業を中止し、ステータスを BLOCKED で報告
- 何が問題で、なぜ自動解決できないかを具体的に記述
- 人間パートナーの判断を待つ

### Retryable（一時的 — 自動リトライ可）

一時的な問題で、リトライにより解決する可能性が高い。

判定基準:
- ツール実行の一時的失敗（ネットワーク、タイムアウト）
- テスト実行の失敗（コードの修正で解決可能）
- ファイル操作の一時的エラー
- コンテキストウィンドウの圧縮による情報欠落
- コンテキスト不足（NEEDS_CONTEXT — 人間に追加情報を求めてからリトライ）
- 外部依存の待ち（MCP サーバー応答、外部 API 等 — 解消後にリトライ）

対応アクション:
- Safety Limits の範囲内で自動リトライ（同一ステップ最大3回）
- コンテキスト不足・外部依存待ちの場合は、何が不足しているかを記述して人間パートナーに伝えてからリトライ
- リトライごとに何を変えたかを記録
- 上限到達で Fatal に昇格（人間にエスカレーション）

### Suspension（一時停止 — 人間承認待ち）

エージェント自身では進められないが、エラーではない。人間承認ゲートに到達した正常な停止。

判定基準:
- 人間承認ゲート到達（サスペンションポイント: [1]要件確定, [2]設計承認, [3]計画承認, [11]コミット前）

対応アクション:
- ステータスを報告し、承認対象の内容を提示
- 人間パートナーの承認または却下を待つ
- 承認後に次ステップへ進む。却下の場合は指示に従って戻る

## エラー分類の判断フロー

```
エラー発生
  ├─ 要件・スコープ・セキュリティに関わる？ → Fatal
  ├─ 一時的な技術的問題・情報不足・外部依存待ち？ → Retryable（リトライ上限内なら自動or情報収集後リトライ）
  ├─ 人間承認ゲート（[1][2][3][11]）に到達？ → Suspension
  └─ 判断できない → Fatal（安全側に倒す）
```

## Status との対応

| エラー分類 | 報告 Status | 状態マシン遷移 |
|-----------|------------|--------------|
| **Fatal** | BLOCKED | failed → suspended（人間エスカレーション） |
| **Retryable** | （リトライ後に DONE or BLOCKED） | failed → in_progress（リトライ） |
| **Suspension** | （承認待ち — ステップに応じた報告） | in_progress → suspended |

## 共通ルール

- 迷ったら Fatal（安全側に倒す）
- Retryable でリトライする場合、前回と同じアプローチを繰り返さない
- エラー分類は完了報告（completion-report-format.md）の障害内容セクションに記載する
