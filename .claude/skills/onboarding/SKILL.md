# Onboarding（ハーネス利用ガイド）

## 概要

ハーネスが導入されたプロジェクトに参加した新メンバーに、ワークフローの使い方を対話的に教えるスキル。

**入力:** なし（対話で進める）
**出力:** メンバーがワークフローを理解し、自律的に使えるようになる

## いつ使うか

- 新しいメンバーがプロジェクトに参加したとき
- ハーネスの使い方を忘れたとき
- ワークフローの全体像を確認したいとき

## 対話フロー

以下のステップを順番に進める。各ステップの最後に「ここまでで質問ある？」と聞き、質問があれば答えてから次に進む。

### Step 1: ハーネスの全体像

まず、このプロジェクトのハーネス構成を確認して伝える。

1. `.claude/` ディレクトリの中身を確認する
2. 以下を説明する:

```
このプロジェクトには AI 駆動開発のためのハーネスが導入されています。

ハーネスは 4 つの仕組みで構成されています:

1. ワークフロー（12ステップ）— 開発の進め方
2. スキル — 各ステップの具体的な手順書
3. エージェント — スキルから呼び出される専門家
4. ルール — 常に適用されるコーディング・テスト・セキュリティの基準

あなたが覚えることは「ワークフローの流れ」と「スキルの呼び出し方」だけです。
エージェントはスキルが自動的に呼び出すので、直接使う必要はありません。
ルールは常に自動適用されるので、意識する必要はありません。
```

### Step 2: ワークフロー

12ステップのワークフローを説明する。

```
開発は以下の 12 ステップで進めます:

[1] 要件理解  — 何を作るか明確にする
[2] 設計      — どう作るか決める
[3] 計画      — 実装手順を決める
[4] 実装      — TDD でコードを書く
[5] テスト    — テストを実行する
[6] リファクタ — コードを簡素化する
[7] 品質テスト — 境界値・エッジケースのテストを追加する
[8] レビュー  — 3 観点（仕様・品質・セキュリティ）でレビューする
[9] 完了検証  — 最終確認する
[10] 整理     — 不要ファイルを片付ける
[11] コミット — 成果をコミットする
[12] 振り返り — セッションを振り返る

全ステップを毎回やる必要はありません。
タスクの規模に応じてスキップできますが、その場合は人間パートナー（あなた）に確認します。
```

メンバーに聞く: 「普段の開発で、この中で特に気になるステップはありますか？」

気になるステップがあれば、そのスキルを先に詳しく説明する。

### Step 3: スキルの使い方

プロジェクトにインストールされているスキル一覧を `.claude/skills/` から読み取って表示する。

```
スキルは各ステップの詳しい手順書です。
Claude に「要件を整理して」「レビューして」と言えば、対応するスキルが自動的に使われます。

明示的に呼び出す場合は、スキル名を伝えてください:
- 「requirements スキルで要件を整理して」
- 「code-review スキルでレビューして」
- 「retrospective で振り返りをして」
```

各スキルの概要を1行ずつ紹介する（description フィールドを読み取る）。

### Step 4: 日常の使い方

実際の開発フローを具体例で説明する。

```
実際の使い方の例:

1. 「ユーザー登録機能を追加して」と指示する
2. Claude がワークフローに沿って進める:
   - まず要件を確認する（Step 1）
   - 設計を提案する（Step 2）
   - 実装計画を立てる（Step 3）
   - TDD でコードを書く（Step 4-5）
   - コードを簡素化する（Step 6）
   - 品質テストを追加する（Step 7）
   - 3 観点でレビューする（Step 8）
   - 最終確認する（Step 9）
   - コミットする（Step 11）
3. 各ステップで「これでいい？」と確認される
4. 指摘があれば修正される

あなたの役割は:
- 要件を伝える
- 各ステップで承認する
- 方向が違ったら修正を指示する
```

### Step 5: モジュール（該当する場合）

`.mcp.json` が存在する場合、導入済みモジュールを説明する。

**Playwright MCP が入っている場合:**
```
このプロジェクトには Playwright MCP モジュールが入っています。
Claude がブラウザを操作して画面を確認できます。

使い方:
- 「この画面を確認して」→ Claude がブラウザで画面を見に行く
- 「E2E テストを作って」→ 画面を観察してからテストコードを生成する

ブラウザ操作は browser-operator エージェントが担当するので、
メインの会話のコンテキストは汚れません。
```

**Figma MCP が入っている場合:**
```
このプロジェクトには Figma MCP モジュールが入っています。
Claude が Figma ファイルを参照・編集できます。

使い方:
- 「Figma のデザインを確認して」→ Claude が Figma を見に行く
- 「この画面のデザインコンテキストを取得して」→ 設計情報を取得する

Figma 操作は figma-operator エージェントが担当します。
初回は OAuth 認証が必要です。
```

### Step 6: 注意点

```
最後に、いくつか大事な注意点:

1. Claude は破壊的な操作（ファイル削除、force push 等）の前に必ず確認します
2. 「違う」「そうじゃない」と言えば、Claude はフィードバックを記録して改善します
3. セッション終了時に振り返りのリマインダーが出ます。時間があれば実行してください
4. わからないことがあれば、いつでも「/onboarding」で使い方を確認できます
```

### Step 7: 質問タイム

```
以上がハーネスの基本的な使い方です。
何か質問はありますか？
```

質問があれば丁寧に答える。質問がなければ終了。

## 動作指針

- **押し付けない**: 一方的に説明を流さない。各ステップで間を取り、質問を受け付ける
- **具体的に**: 抽象的な説明ではなく、「こう言えばこう動く」を示す
- **プロジェクト固有**: `.claude/` の実際の中身を読んで、このプロジェクトに入っているスキル・モジュールだけを説明する
- **短く**: 各ステップの説明は簡潔に。詳しく知りたいと言われたら深掘りする

## 委譲指示

**このスキルは委譲しない。** メインセッションが直接対話する。
エージェントは不要 — 人間との対話が目的なので、サブエージェントに任せると体験が損なわれる。
