# Onboarding（ハーネス利用ガイド）

## 概要

ハーネスが導入されたプロジェクトに参加した新メンバーに、ワークフローの使い方を対話的に教えるスキル。

**入力:** なし（対話で進める）
**出力:** メンバーがワークフローを理解し、自律的に使えるようになる

## いつ使うか

- 新しいメンバーがプロジェクトに参加したとき
- ハーネスの使い方を忘れたとき
- ワークフローの全体像を確認したいとき

## 対話フロー

2つのフェーズに分かれている。Phase A だけで使い始められる。Phase B は必要になったら実行する。

---

### Phase A: クイックスタート（15分）

最低限知っておくべきことだけ伝える。これだけで使い始められる。

#### Step 1: ハーネスとは

```
このプロジェクトには AI 駆動開発のためのハーネスが導入されています。

覚えることは 2 つだけ:

1. 開発は「12ステップのワークフロー」に沿って進む
2. Claude に指示すれば、対応する「スキル」が自動的に使われる

エージェント（専門家 AI）やルール（品質基準）は裏で自動的に動くので、意識する必要はありません。
```

#### Step 2: ワークフロー概要

```
開発は以下の流れで進みます:

[1] 要件理解 → [2] 設計 → [3] 計画
→ [4] 実装 → [5] テスト → [6] リファクタ → [7] 品質テスト → [8] レビュー
→ [9] 完了検証 → [10] 整理 → [11] コミット → [12] 振り返り

全ステップを毎回やる必要はありません:
- typo 修正 → [4] 実装 → [11] コミット だけ
- バグ修正 → [4] 実装 → [8] レビュー → [11] コミット
- 新機能  → 全12ステップ

規模の判定は Claude が確認します。
```

#### Step 3: 使い始め方

```
使い方はシンプルです:

- 「ユーザー登録機能を追加して」→ Claude がワークフローに沿って進める
- 「このバグを直して」→ 規模を判定して適切なステップで進める

各ステップで「これでいい？」と確認されます。
方向が違ったら「違う」「そうじゃない」と言えば修正されます。

スキルの一覧は .claude/skills/README.md にあります。
```

メンバーに聞く: 「ここまでで質問ある？なければ、さっそく最初のタスクをやってみましょう。」

質問がなければ **Phase A 完了**。実際にタスクを進めながら覚える。

---

### Phase B: 詳細ガイド（30分）

Phase A で使い始めた後、もっと詳しく知りたくなったら実行する。

#### Step 4: スキル詳細

プロジェクトにインストールされているスキル一覧を `.claude/skills/README.md` から読み取って表示する。

```
スキルは各ステップの詳しい手順書です。
Claude に「要件を整理して」「レビューして」と言えば、対応するスキルが自動的に使われます。

明示的に呼び出す場合は、スキル名を伝えてください:
- 「requirements スキルで要件を整理して」
- 「code-review スキルでレビューして」
- 「retrospective で振り返りをして」
```

各スキルの概要を1行ずつ紹介する（description フィールドを読み取る）。

メンバーに聞く: 「普段の開発で、この中で特に気になるステップはありますか？」

気になるステップがあれば、そのスキルを先に詳しく説明する。

#### Step 5: モジュール（該当する場合）

`.mcp.json` が存在する場合、導入済みモジュールを説明する。

**Playwright MCP が入っている場合:**
```
このプロジェクトには Playwright MCP モジュールが入っています。
Claude がブラウザを操作して画面を確認できます。

使い方:
- 「この画面を確認して」→ Claude がブラウザで画面を見に行く
- 「E2E テストを作って」→ 画面を観察してからテストコードを生成する

ブラウザ操作は browser-operator エージェントが担当するので、
メインの会話のコンテキストは汚れません。
```

**Figma MCP が入っている場合:**
```
このプロジェクトには Figma MCP モジュールが入っています。
Claude が Figma ファイルを参照・編集できます。

使い方:
- 「Figma のデザインを確認して」→ Claude が Figma を見に行く
- 「この画面のデザインコンテキストを取得して」→ 設計情報を取得する

Figma 操作は figma-operator エージェントが担当します。
初回は OAuth 認証が必要です。
```

#### Step 6: 裏側の仕組み

```
ハーネスの裏側には以下の仕組みがあります:

- エージェント — スキルから自動的に呼び出される専門家 AI（一覧は .claude/agents/README.md を参照）
  - 例: implementer（実装）、security-reviewer（セキュリティレビュー）
  - レビュアーは読み取り専用。実装者のコードを壊せない設計

- ルール — 常に自動適用される品質基準（一覧は .claude/rules/README.md を参照）
  - コーディング規約、セキュリティ、テスト方針、Git 運用、ドキュメント、フィードバック記録、ワークフロー

- フック — イベント駆動の自動化
  - 例: メインセッションがコードを書こうとしたらブロック（エージェントに任せる設計）
  - 例: コミット前に検証証拠（last-verification.json）が必要（verification-gate）
  - 例: セッション終了時に振り返りリマインダーを表示（session-end-retrospective）

普段は意識する必要はありません。興味があれば .claude/ の中を見てください。
```

#### Step 7: 注意点と質問

```
最後に、いくつか大事な注意点:

1. Claude は破壊的な操作（ファイル削除、force push 等）の前に必ず確認します
2. 「違う」「そうじゃない」と言えば、Claude はフィードバックを記録して改善します
3. セッション終了時に振り返りのリマインダーが出ます。時間があれば実行してください
4. わからないことがあれば、いつでも「/onboarding」で使い方を確認できます
```

```
以上がハーネスの詳細な仕組みです。
何か質問はありますか？
```

質問があれば丁寧に答える。質問がなければ終了。

## 動作指針

- **押し付けない**: 一方的に説明を流さない。各ステップで間を取り、質問を受け付ける
- **具体的に**: 抽象的な説明ではなく、「こう言えばこう動く」を示す
- **プロジェクト固有**: `.claude/` の実際の中身を読んで、このプロジェクトに入っているスキル・モジュールだけを説明する
- **短く**: 各ステップの説明は簡潔に。詳しく知りたいと言われたら深掘りする

## 委譲指示

**このスキルは委譲しない。** メインセッションが直接対話する。
エージェントは不要 — 人間との対話が目的なので、サブエージェントに任せると体験が損なわれる。
