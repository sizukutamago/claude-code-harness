# コア概念ガイド

claude-code-harness の設計思想と各構成要素の「なぜ」を説明するドキュメント。

- **architecture-design.md** は「何を作るか・どう構成するか」
- **harness-development.md** は「どうやって拡張・開発するか」
- **このドキュメント** は「なぜこういう仕組みなのか」

チームに新しく参加したメンバーはここから読むと、構造の背景にある意図を掴める。

---

## 1. コーディネーターパターン

メインセッション（Claude Code を直接操作しているセッション）はコードを書かない。

### なぜか

LLM は「コードを書くな」と指示されても、プレッシャーや文脈によって指示を無視することがある。「信頼してほしい」と言っても信頼性は生まれない。**Structure over Hope** — 構造で強制するほうが安全性は高い。

### どう強制するか

`coordinator-write-guard` フック（PreToolUse）が、メインセッションによるファイル書き込みをブロックする。指示があっても物理的に実行できない状態にする。

### メインセッションの責務

- 人間パートナーとの対話（要件確認・承認依頼）
- スキルの起動とエージェントへのタスク委譲
- エージェントの出力レビューと次ステップの決定

コードを書く・テストを実行する・ファイルを直接編集する、といった作業はすべてエージェントが担う。

詳細は `docs/design/architecture-design.md` の「コーディネーター設計」セクションを参照。

## 2. エージェントディスパッチ

スキルはエージェントにタスクを「委譲」する。エージェントは独立したサブプロセスとして動作し、完了後にステータスを報告する。

### 最小権限（tools フロントマター）

エージェントはフロントマターの `tools` フィールドで使えるツールが制限される。たとえばレビュアーエージェントは読み取り専用ツールしか持たない。実装エージェントだけが書き込みを許可される。

これにより「レビュー中に誤ってコードを書き換える」ような事故を構造的に防ぐ。

### コンテキスト注入パターン

エージェントにファイルを読ませるのではなく、dispatch 時のプロンプトに必要な情報をすべて埋め込む。

理由: エージェントがファイルを自律的に読み始めると、何を読んだかが不透明になり、再現性が下がる。プロンプトに埋め込む方式なら、エージェントに渡したコンテキストが明示的になる。

### ステータス報告

エージェントは処理完了後に標準化されたステータスを返す:

| ステータス | 意味 |
|-----------|------|
| DONE | 正常完了 |
| DONE_WITH_CONCERNS | 完了したが懸念あり |
| NEEDS_CONTEXT | 情報不足で続行不能 |
| BLOCKED | 技術的・制約的理由で続行不能 |

コーディネーターはこのステータスをもとに次のアクションを決定する。

詳細は `.claude/agents/_shared/` の各リファレンスを参照。

## 3. スキルパイプライン

12ステップのワークフローは、各スキルが「入力を受け取り、出力を生成し、次のスキルに渡す」パイプラインとして設計されている。

### 各ステップの入力→出力の連鎖

```
[1] 要件理解  → requirements.md, context.md
[2] 設計      → specs/, adr/, decisions/
[3] 計画      → plans/
[4] 実装      → 実装コード
[5] テスト    → テストコード
[6] リファクタ → 改善後のコード
[7] 品質テスト → 追加テストケース
[8] レビュー  → レビュー指摘・承認
[9] 完了検証  → 検証レポート
[10] 整理     → クリーンアップ済み状態
[11] コミット  → git commit
[12] 振り返り  → session-feedback.jsonl への記録・改善提案
```

### Integration セクションによる依存宣言

各スキルの末尾には `## Integration` セクションがある。前提スキル・必須ルール・次のステップ・逆方向の依存を宣言することで、スキル間の関係が明示的になる。

### タスク規模別のパス

毎回12ステップ全部を実行する必要はない。タスクの規模によってショートカットパスが用意されている:

| 規模 | 適用ステップ |
|------|------------|
| Tiny | [4] 実装 → [11] コミット |
| Small | [4] 実装 → [8] レビュー → [11] コミット |
| Normal | 全12ステップ |
| Large | roadmap でフェーズ分割 → 各フェーズを Normal で回す |

詳細は `.claude/rules/workflow.md` を参照。

## 4. フックシステム

フックは「何かが起きたとき（ツール実行前・後・拒否）」に自動実行されるスクリプト。ワークフローの外側から品質・安全性を担保する構造的ガードとして機能する。

### PreToolUse: 事前ブロック

ツールが実行される前に介入し、危険な操作を止める。

| フック | 目的 |
|--------|------|
| coordinator-write-guard | メインセッションによるファイル書き込みをブロック |
| secret-scanner | シークレット（APIキー等）のコミットをブロック |
| verification-gate | 検証なしの完了宣言をブロック |

### PostToolUse: 操作後ログ

ツール実行後に自動でログを記録する。

| フック | 目的 |
|--------|------|
| post-tool-log | ファイル編集・書き込み操作のツール実行ログを記録 |
| workflow-event-logger | Agent dispatch のイベントログを記録 |

### PermissionDenied: 拒否の記録

ユーザーがツール実行を拒否したとき、その記録を残す。後の振り返りで「なぜ止められたか」を分析できる。

### 設計原則

- **セキュリティフックは fail-closed**: ブロック判定できなければ実行を止める。誤検知より見落としのほうがコストが高い
- **ログフックは fail-open**: 記録に失敗しても処理は続行する。記録の失敗で本来の作業を止めない

詳細は `.claude/hooks/` を参照。

## 5. フィードバックループ

ハーネスは「使うほど改善される」仕組みを持つ。改善は即時記録→定期分析→テンプレート還元のサイクルで回る。

### 即時記録: session-feedback.jsonl

ユーザーが指摘・修正・方向転換を行ったとき、フックとルールが協調して `.claude/harness/session-feedback.jsonl` に自動記録する。「後でまとめて」ではなく、指摘された瞬間に書く。

### 振り返り: retrospective スキル

セッション終了時や定期的なタイミングで retrospective スキルを実行する。session-feedback.jsonl を分析し、繰り返しパターンと改善提案を生成する。

### 定量化: eval による効果測定

`eval/` ディレクトリに promptfoo ベースの評価セットがある。ハーネスの変更前後で AI の行動がどう変わったかを定量的に測定できる。「改善した気がする」ではなく証拠ベースで判断する。

### 還元: harness-contribute

プロジェクト側で発見した改善をハーネスのテンプレートリポジトリに PR として反映するスキル。個別プロジェクトの知見がチーム全体に伝播する。

詳細は `.claude/skills/retrospective/SKILL.md`、`docs/guides/harness-development.md` を参照。

## 6. エラー分類と状態マシン

エージェントとワークフローは、エラーを3種類に分類して処理する。分類を統一することで、コーディネーターが一貫したハンドリングを行える。

### Fatal / Retryable / Suspension の3分類

| 分類 | 意味 | 対応 |
|------|------|------|
| Fatal | 自動リトライ不可。根本原因の解決が必要 | 人間パートナーへエスカレーション |
| Retryable | 一時的なエラー。再試行で解消する可能性がある | Safety Limits の範囲内で自動リトライ |
| Suspension | エラーではないが進行不能 | 人間承認ゲート到達時の一時停止 |

### ステップの状態遷移

各ワークフローステップは以下の状態を持つ:

```
pending → in_progress → completed
                      → failed (Fatal)
                      → suspended (Suspension)
```

### Safety Limits による安全装置

リトライ上限・連続失敗上限を設けることで、エージェントが無限ループに陥らないようにする。上限に達したら自動的に Suspension に遷移し、人間に制御を戻す。

詳細は `.claude/agents/_shared/error-classification.md` を参照。
