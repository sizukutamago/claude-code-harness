# 多層観察アーキテクチャ — Shape Up Pitch

## 問題 (Problem)

Claude Code を駆使したハーネスで実装したプロダクト（例: EC サイト）に対して、**実装エージェント自身の振る舞い・成果物の品質・ワークフロー遵守を、第三者目線で継続的に観察する仕組みが存在しない**。

結果として:

- エージェントが暴走・ループ・ワークフロー巧妙スキップをしても誰も気づかない
- 寝てる間に動かしたくても何が起きたか再現できず安心して任せられない
- プロダクトが「コードは動く」だけで UIUX・仕様準拠・デザインとして成立してるか不明
- ハーネスとして実装した機能自体が「動作する形」になってるか分からない
- 現在の回避策は **「何もしていない（運を信じている）」** — ノーガード状態

そしてこれらの痛みは、ハーネス開発者・チームメンバー・エンドユーザーという**多層の顧客**に、それぞれ別の形で現れる。

## Essential Intent

**多層観察アーキテクチャを実現する。**

実装層 → 監視層 → メタ監視層の3層構造で、エージェントの振る舞いと成果物を継続的に観察し、見落とし・自己盲点を仕組みで補い、観察の観点自体を時間とともに進化させる。

## Appetite

**Full（Tier A+B+C+基盤強化、課題11件）**

段階的に4 Phase に分割してボトムアップで構築する。

## 解決策の骨格 (Solution Sketch)

### アーキテクチャ（4層）

```
┌────────────────────────────────────────────┐
│ L3: メタ監視層                              │
│  - 神エージェント                            │
│  - 監視観点の発見→L2 反映                  │
└────────────────────────────────────────────┘
              ↑ 監視対象として渡す
┌────────────────────────────────────────────┐
│ L2: 監視層（第三者目線エージェント群）       │
│  - プロダクトユーザー目線エージェント        │
│    (UIUX / 仕様準拠 / デザイン)             │
│  - ハーネスユーザー目線エージェント          │
│    (ワークフロー違反 / 使いにくさ)           │
│  - プロセス監視エージェント                  │
│    (暴走 / ループ / 暗黙仮定)                │
└────────────────────────────────────────────┘
              ↑ 成果物・ログを渡す
┌────────────────────────────────────────────┐
│ L1: 実装層（観察される側）                   │
│  - Claude Code (ハーネス導入済み)           │
│  - RALPH Runner v1 (タスク実装駆動)         │
│  - 各種スキル・エージェント                  │
└────────────────────────────────────────────┘
              ↑ 駆動
┌────────────────────────────────────────────┐
│ L0: 駆動基盤                                │
│  - snarktank/ralph (メタループ駆動)         │
│  - EC サイト dogfood target                 │
│  - Claude Code 常駐の仕組み                  │
│  - リセット・リカバリ自動化                  │
└────────────────────────────────────────────┘
```

**重要な区別:**

- **ハーネス内蔵の RALPH Runner v1**: 個別タスク実装を TDD サイクルで駆動する（既存）
- **snarktank/ralph**: 寝てる間のメタループ駆動（EC 作成 → ハーネス改善 → リセット → 繰り返し）— 未組み込み

### Phase 分割（ボトムアップ）

#### Phase 1: L0 駆動基盤（MVP の土台）

**対象課題:** L0-1, L0-2, L0-3, L0-5, L0-6

**成果物:**
- `runner/meta-loop/` — snarktank/ralph 組み込み
- `samples/ec-sample/` — dogfood 対象 EC サイトプロジェクトの雛形
- Claude Code 常駐起動スクリプト（launchctl or systemd）
- 完成判定条件の明文化（AC ベース）とリセットスクリプト
- 失敗時の自動リカバリ（RALPH 監視）

**完了条件:** EC サイトの雛形に対してメタループを1周回せる

#### Phase 2: L1 実装層の可観測性

**対象課題:** L1-1, L1-2, L1-4

**成果物:**
- エージェント実行トレース収集 hook（post-tool-log の拡張）
- ワークフロー遵守の構造化ログ（workflow-events.jsonl の拡張）
- ハーネス機能テストスイート（ハーネス自身の機能が動作するかの自己検証）

**完了条件:** 任意の時点のセッションログから、エージェントが何をしたか・ワークフローを守ったかを第三者が再構成できる

#### Phase 3: L2 監視層（二目線）

**対象課題:** L2-1, L2-2

**成果物:**
- `.claude/agents/product-user-reviewer.md` — プロダクトユーザー目線エージェント（Read only + Playwright MCP + Figma MCP）
- `.claude/agents/harness-user-reviewer.md` — ハーネスユーザー目線エージェント（Read only）
- 監視出力の集約先（`.claude/harness/observation-log.jsonl`）と次セッション注入プロトコル

**完了条件:** 1セッション実行後、2目線から観察結果が自動生成され、次セッションの初期コンテキストに注入される

#### Phase 4: L3 メタ監視層（神エージェント v1）

**対象課題:** L3-5（「観点発見→監視層反映」）

**成果物:**
- `.claude/agents/meta-observer.md` — 神エージェント（Read only）
- 監視観点レジストリ（`.claude/harness/observation-points.yaml`）
- 観点の追加・非推奨化のワークフロー（人間承認ゲート付き）

**完了条件:** 神エージェントが L2 エージェントのレビューを行い、「この観点が足りない」と提案し、人間承認を経て L2 エージェントのプロンプトに反映される

### 解決しないこと (No Goals)

- **L3 のフル機能**（L3-1/L3-2/L3-6/L3-7）: 自己盲点対策、観点陶汰、判断基準は Phase 4 では扱わない。Phase 4 で「つなぎ層」の骨格を作り、運用データが溜まってから拡張する
- **L2-3/L2-4**: プロセス監視と監視出力の扱い定義は Phase 2+3 の運用で見えてから次フェーズで扱う
- **L1-3**（寝てる間の再現）と **L0-4**（人間介在削減）: L0+L1 が固まれば半自動で解決する領域
- **リアルタイム介入**: 監視は観察と提案に留め、エージェントの実行中に割り込まない

## トレードオフ（受容済み）

| トレードオフ | 影響 |
|------------|-----|
| API/トークンコスト増加 | 監視層分 + メタ監視層分でコストが倍増以上。L0-3（コスト上限）は将来課題として残る |
| 実プロジェクト開発の遅延 | ハーネス改善に集中するため、flow-pilot/kondate 等の実案件が遅延する |
| 複雑さの管理 | 4層構造は認知負荷が高い。各層の責務境界を明文化しないと崩壊する |
| dogfood の現実性不足 | EC サイト雛形は現実の実案件ほど多様でない。発見できる問題に偏りが出る |

## リスクと緩和

| リスク | 緩和策 |
|------|--------|
| 監視エージェントが的外れな指摘ばかりする | Phase 3 の完了条件に「次セッションで実際に使えるか」の検証を含める |
| snarktank/ralph の組み込みが想定以上に重い | Phase 1 の appetite を先に切り、1週間で動かなければスキップして Phase 2 に進む |
| 4層の複雑さで自滅する | 各 Phase 完了後に必ず振り返りセッションを入れ、各層の責務を再確認する |
| EC サンプルが「リアルな痛み」を生まない | Phase 2 終了後に dogfood target を複数用意（EC / CLI tool / Web API）検討 |
| 神エージェント自身が暴走する | Phase 4 では神エージェントは**提案のみ**、L2 エージェントの変更は必ず人間承認を通す |

## 次のアクション

1. このピッチの承認を得る
2. 承認後、Phase 1（L0 駆動基盤）を `requirements/REQ-XXX-meta-loop-foundation/` として要件定義に進む
