# 0010: 外部オーケストレーターの導入（RALPH Runner）

**Status:** Approved
**Date:** 2026-04-10

## 背景

現行ハーネスのワークフロー制御は、AI セッション内のプロンプト駆動ディスパッチ（start-workflow スキル → Agent tool）で行われている。この設計には3つの構造的限界がある:

1. **AI が AI を管理している** — プロンプトの指示力に依存し、無視されたら止める手段がない
2. **コンテキストウィンドウが単一障害点** — 12ステップを1セッションで走ると後半でコンテキストが劣化する
3. **自己改善が人間ボトルネック** — 学習のセッション間持ち越し機構がない

RALPH（https://github.com/snarktank/ralph）の外部オーケストレーターパターンを参考に、制御を AI の外に出すことで、プロンプトの指示力ではなくアーキテクチャの強制力で品質を担保する方針に転換する。

## 選択肢

### 選択肢 A: Bash スクリプト
- 概要: `ralph-runner.sh` がループを制御し、`claude -p` を毎ステップ fresh spawn する
- メリット: シンプル、透明、依存なし、RALPH と同じ実績あるパターン
- デメリット: 複雑なエラーハンドリングが苦手、JSON 操作に jq 依存

### 選択肢 B: Node.js スクリプト
- 概要: `ralph-runner.mjs` が `child_process.spawn` で `claude -p` を呼ぶ
- メリット: JSON 操作が自然、エラーハンドリングが容易、既存フックと同じランタイム
- デメリット: Bash より抽象度が高く、デバッグ時の透明性が落ちる

### 選択肢 C: Claude Code SDK (Agent SDK)
- 概要: Anthropic 公式の SDK でプログラマティックにセッションを管理
- メリット: イベントストリーム・ツール承認・セッション管理が組み込み。最も強力
- デメリット: SDK の安定性・学習コストが未知数。ブラックボックス度が高い

## 決定

**選択肢 A: Bash スクリプト** を採用する。

理由:
- RALPH で実績のあるパターン。シンプルさが最大の武器
- 外部オーケストレーターの本質は「ループ + claude -p + 品質チェック」であり、Bash で十分
- jq は macOS/Linux に標準で利用可能（追加依存なし）
- 透明性が高く、デバッグ・カスタマイズが容易
- 将来的に Node.js や SDK に移行する場合も、インターフェース（plan.json, learnings.jsonl）は同じなので移行コストが低い

## 影響

- `runner/` ディレクトリを新設し、オーケストレーター関連スクリプトを配置
- `planning` スキルに plan.json 出力機能を追加（既存の plan.md と並行）
- `.claude/harness/` に plan.json, learnings.jsonl, runs/ を追加
- 既存のスキル・エージェント・フック・ルールは変更不要
- start-workflow スキルは対話フェーズ用に残し、マシンフェーズは RALPH Runner が担当
