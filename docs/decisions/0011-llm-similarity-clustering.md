# 0011: review-memory の類似度判定に LLM 推論を使う

**Status:** Approved
**Date:** 2026-04-11
**Covers:** REQ-001

## 背景

review-memory スキルは、コードレビューの指摘を `review-findings.jsonl` に蓄積し、同じアンチパターンが繰り返し出現したら `review-conventions.md` に昇格させる。この「同じアンチパターン」の判定方法を決める必要がある。

RALPH Runner v1 の `conventions-builder.sh` は文字列の完全一致で判定している。しかし code-review の指摘は自然言語で記述されるため、本質的に同じアンチパターンが異なる表現で記録される可能性が高い。

例:
- 「正規表現 `content=\"[^\"]*\"` のパターンは引用符を含む content で破綻する」
- 「`[^\"]*` のような正規表現では `"` を含む値が抽出できない」

これらは同じアンチパターン（脆弱な正規表現パーサ）だが、文字列としては完全一致しない。

## 選択肢

### 選択肢 A: 完全一致
- 概要: `category` と `pattern` の文字列完全一致で判定
- メリット: 実装が簡単（`runner/lib/conventions-builder.sh` と同じロジック）、LLM コストなし、決定的
- デメリット: 自然言語の表現揺れで同じパターンが別クラスタになる。昇格が発生しにくい

### 選択肢 B: キーワードマッチング
- 概要: `category` の一致 + `pattern` のキーワード集合の Jaccard 類似度が閾値以上
- メリット: LLM コストなし、ある程度の表現揺れに対応
- デメリット: キーワード抽出のルールを書く必要がある、閾値チューニングが必要、日本語の形態素解析が必要

### 選択肢 C: Embedding ベースの類似度
- 概要: 各 pattern を埋め込みベクトル化し、cosine 類似度で判定
- メリット: 意味的な類似度を捉えられる、決定的
- デメリット: 埋め込みモデルの API 呼び出しが必要、閾値チューニングが必要、埋め込みキャッシュが必要

### 選択肢 D: LLM 推論（review-memory-curator エージェント）
- 概要: Claude エージェント（review-memory-curator）が新規指摘と既存クラスタ代表を比較して判定する
- メリット: 意味的な類似度を最も柔軟に判定できる、プロンプトエンジニアリングでチューニング可能、追加の依存なし（既に Claude を使っている）
- デメリット: LLM 呼び出しコスト、非決定性、プロンプト設計が必要

## 決定

**選択肢 D: LLM 推論（review-memory-curator エージェント）** を採用する。

### 理由

1. **意味的類似度の柔軟性**: code-review の指摘は自然言語で記述されるため、完全一致やキーワードマッチでは見落としが多い。LLM は文脈を理解できる
2. **既に Claude を使っている**: 追加の依存（埋め込みモデル API、形態素解析ライブラリ等）が不要
3. **プロンプトでチューニング可能**: 判定基準を自然言語で記述できるため、チューニングが柔軟
4. **代表エントリとの比較でコスト抑制**: 新規指摘ごとにクラスタ代表 1件だけを渡すので、LLM 呼び出し回数は N+1 ではなく M（新規指摘数）に抑えられる

### コスト対策

- **代表エントリのみ渡す**: `category + pattern` で既にクラスタ化された既存エントリの代表 1件だけを curator に渡す。全エントリを渡さない
- **新規指摘ごとに 1 回**: バッチ呼び出しは出力構造が複雑になるため避ける。新規指摘ごとに単純な 1 回呼び出しにする（並列化可能）
- **初回マイグレーションのみバッチ**: 既存9件を一括でクラスタリングする初回マイグレーションは特別扱い（1回の LLM 呼び出しで全件処理）

## 結果

### 影響
- `.claude/agents/review-memory-curator.md` を新設する（Read only、model: opus）
- curator の出力は `{"cluster_id": "c-XXX" | null}` の JSON 単一値
- curator のエラー時は「新規クラスタ」として扱う（フォールバック）
- 1 回の code-review で新規指摘 M 件 × 1 回の LLM 呼び出し = M 回（並列可能）
- 初回マイグレーション時のみ、既存9件を一括でクラスタリングする特殊処理が発生

### 制約
- curator の判定が非決定的なため、同じ入力で異なる結果が返る可能性がある（通常は同じ結果になるはずだが保証はない）
- curator のプロンプト・判定基準の変更は、過去のクラスタリング結果と不整合を起こす可能性がある

### 将来の検討事項
- 大規模化した場合、Embedding ベース（選択肢 C）との併用も検討できる
- 「類似度スコア付き」で返す形式（`{"cluster_id": "c-001", "confidence": 0.85}`）も将来的には検討可能
